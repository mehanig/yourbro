# Shared Pages: Email-Based Access Control via API-Signed Identity Tokens

## Problem

Currently pages have two access levels: **paired** (full access) and **anonymous** (public pages only). There's no middle ground. A page owner can't say "let `friend@gmail.com` see this page" without that person pairing with the agent.

## Goal

Allow page owners to grant access to specific Google accounts. A logged-in yourbro user who visits a page gets verified identity passed to the agent, which checks it against an allow-list in `page.json`. A per-page access code provides a second layer of defense that is fully under the agent's control.

## Threat Model

Two independent threats to shared page confidentiality:

1. **Identity spoofing** — an attacker claims to be `friend@gmail.com` without owning that account. Solved by the API-signed identity token (Ed25519 JWT).
2. **Compromised API server** — an attacker who controls the API can forge identity tokens (they have the Ed25519 private key). Solved by the per-page access code, which the server never sees (it travels only inside the E2E encrypted payload).

Both layers must pass for shared page access. Either layer alone is insufficient.

## Design: Two-Layer Shared Access

### Layer 1: API-Signed Identity Token (email verification)

The API is the only party that has verified the Google login. It issues a short-lived Ed25519-signed JWT ("identity token") that the browser includes inside the E2E encrypted relay payload. The agent verifies the signature using the API's public key.

#### Why Ed25519 (not HMAC)

The current session JWTs use HMAC-SHA256 with `JWT_SECRET`. That's a symmetric secret — sharing it with agents would let any agent forge session tokens. Ed25519 is asymmetric: the API signs with a private key, agents verify with the public key. No secret leaves the API.

### Layer 2: Per-Page Access Code (server-blind secret)

Each shared page has a random access code generated by the agent at page creation time and stored in `page.json`. The page owner shares this code out-of-band (chat, email, etc.) with invited viewers. The viewer enters it once per page; the browser stores it in IndexedDB and includes it in every E2E encrypted relay request for that page.

**The API server never sees this code.** It exists only in:
- `page.json` on the agent's filesystem
- The viewer's IndexedDB
- The E2E encrypted payload (opaque to the server)

Even if the API is fully compromised and the attacker can forge identity tokens for any email, they still can't access shared pages without the access code.

### Combined Flow

```
SETUP (page owner via OpenClaw):

  OpenClaw writes page.json:
  {
    "title": "My Page",
    "allowed_emails": ["friend@gmail.com"],
    "access_code": "A7X3KP9M"        ← agent generates, 8 alphanumeric chars
  }

  Owner sends to friend: "Page is at yourbro.ai/p/me/mypage, code is A7X3KP9M"


VIEWING (friend@gmail.com):

Browser (shell.html)              API (api.yourbro.ai)              Agent
   │                                │                                  │
   │── GET /api/identity-token ────>│                                  │
   │   (yb_session cookie)          │── Verify session                 │
   │                                │── Sign Ed25519 JWT:              │
   │                                │   { email, sub, username, exp }  │
   │<── { token: "eyJ..." } ───────│                                  │
   │                                │                                  │
   │  Prompt: "Enter access code"   │                                  │
   │  (or load from IndexedDB)      │                                  │
   │                                │                                  │
   │── E2E encrypt({               │                                  │
   │     method: GET,               │                                  │
   │     path: /api/page/slug,     │                                  │
   │     headers: {                 │                                  │
   │       X-Identity-Token: eyJ..  │  ← Layer 1 (email proof)        │
   │       X-Access-Code: A7X3KP9M  │  ← Layer 2 (page secret)       │
   │     }                          │                                  │
   │   }) ─────────────────────────>│── blind relay ──────────────────>│
   │                                │   (can't see any headers)        │── E2E decrypt
   │                                │                                  │── Verify identity token sig
   │                                │                                  │── Extract email
   │                                │                                  │── Check: email in allowed_emails?
   │                                │                                  │── Check: access_code matches?
   │                                │                                  │── Both pass → serve page
   │<── E2E encrypted response ────│<─────────────────────────────────│
```

### Security Properties

- **Zero-knowledge preserved**: Both the identity token and access code travel inside the E2E encrypted payload. The API relay never sees either.
- **Defense in depth**: Compromised API (forged identity tokens) is blocked by access code. Stolen access code (e.g., forwarded message) is blocked by email verification.
- **Unforgeable identity**: Ed25519 signature — only the API's private key can produce valid tokens.
- **Short-lived tokens**: 5-minute expiry. No revocation needed.
- **Agent sovereignty**: The agent generates and owns the access code. The API has no role in it.
- **Graceful degradation**: Missing token or code → fall through to paired/public tiers. Existing pages are unaffected.

### Access Decision Matrix

| Viewer | `public: true` | `allowed_emails` + code match | Paired | Private (default) |
|--------|:-:|:-:|:-:|:-:|
| Anonymous (no account) | Yes | No | No | No |
| Logged in, no code | Yes | No | No | No |
| Logged in, wrong code | Yes | No | No | No |
| Has code, wrong email | Yes | No | No | No |
| Logged in + email match + correct code | Yes | **Yes** | No | No |
| Paired user | Yes | Yes | Yes | Yes |

### Phase 1: API — Ed25519 Key Pair & JWKS Endpoint

**Files to create/modify:**

#### 1.1 Generate Ed25519 keypair

Add env var `IDENTITY_SIGNING_KEY` (base64-encoded Ed25519 private key, 64 bytes). Generate once during initial setup. The API loads it at startup.

**File: `api/internal/auth/identity.go`** (new)

```go
package auth

import (
    "crypto/ed25519"
    "encoding/base64"
    "fmt"
    "os"
    "time"

    "github.com/golang-jwt/jwt/v5"
)

type IdentitySigner struct {
    PrivateKey ed25519.PrivateKey
    PublicKey  ed25519.PublicKey
    KeyID      string // base64url of first 8 bytes of public key
}

type IdentityClaims struct {
    Email    string `json:"email"`
    Username string `json:"username"`
    jwt.RegisteredClaims
}

func NewIdentitySigner() (*IdentitySigner, error) {
    keyB64 := os.Getenv("IDENTITY_SIGNING_KEY")
    if keyB64 == "" {
        return nil, fmt.Errorf("IDENTITY_SIGNING_KEY not set")
    }
    keyBytes, err := base64.StdEncoding.DecodeString(keyB64)
    if err != nil || len(keyBytes) != ed25519.PrivateKeySize {
        return nil, fmt.Errorf("invalid IDENTITY_SIGNING_KEY")
    }
    priv := ed25519.PrivateKey(keyBytes)
    pub := priv.Public().(ed25519.PublicKey)
    kid := base64.RawURLEncoding.EncodeToString(pub[:8])
    return &IdentitySigner{PrivateKey: priv, PublicKey: pub, KeyID: kid}, nil
}

func (s *IdentitySigner) SignIdentityToken(email, username string, userID int64) (string, error) {
    now := time.Now()
    claims := IdentityClaims{
        Email:    email,
        Username: username,
        RegisteredClaims: jwt.RegisteredClaims{
            Subject:   fmt.Sprintf("%d", userID),
            Issuer:    "yourbro",
            IssuedAt:  jwt.NewNumericDate(now),
            ExpiresAt: jwt.NewNumericDate(now.Add(5 * time.Minute)),
        },
    }
    token := jwt.NewWithClaims(jwt.SigningMethodEdDSA, claims)
    token.Header["kid"] = s.KeyID
    return token.SignedString(s.PrivateKey)
}
```

### Phase 2: Agent — Access Code Generation, Identity Verification & ACL

**Files to create/modify:**

#### 2.1 Access code generation

**Modify: `agent/internal/handlers/pages.go`** — `readPageMeta()` and page creation logic

When OpenClaw writes a `page.json` with `allowed_emails` but no `access_code`, the agent auto-generates one on first read and writes it back:

```go
func (h *PagesHandler) ensureAccessCode(slug string, meta *pageMeta) {
    if len(meta.AllowedEmails) > 0 && meta.AccessCode == "" {
        meta.AccessCode = generateAccessCode() // 8 uppercase alphanumeric chars
        writePageMeta(slug, meta)              // write back to page.json
    }
}

func generateAccessCode() string {
    // crypto/rand, 8 chars from [A-Z0-9]
    const charset = "ABCDEFGHJKLMNPQRSTUVWXYZ23456789" // no 0/O/1/I ambiguity
    code := make([]byte, 8)
    for i := range code {
        n, _ := rand.Int(rand.Reader, big.NewInt(int64(len(charset))))
        code[i] = charset[n.Int64()]
    }
    return string(code)
}
```

Alternatively, OpenClaw can set the access code explicitly in `page.json`. The agent just uses whatever is there.

The agent should log the access code when it generates one:
```
=== ACCESS CODE for page "mypage": A7X3KP9M ===
Share this code with invited viewers.
```

#### 2.3 Extract identity and access code in relay router

**Modify: `agent/internal/relay/router.go`**

After E2E decryption of the inner request, extract both headers:

```go
// After decrypting inner request (existing code)

// Layer 1: Identity token (email verification)
if tokenStr := innerReq.Headers["X-Identity-Token"]; tokenStr != "" {
    claims, err := r.IdentityVerifier.Verify(tokenStr)
    if err == nil {
        ctx = handlers.WithIdentityEmail(ctx, claims.Email)
        ctx = handlers.WithIdentityUsername(ctx, claims.Username)
    }
    // If verification fails, silently ignore — treat as anonymous.
    // Do NOT reject the request. The viewer might still be paired or viewing a public page.
}

// Layer 2: Access code (server-blind secret)
if code := innerReq.Headers["X-Access-Code"]; code != "" {
    ctx = handlers.WithAccessCode(ctx, code)
}
```

#### 2.4 Update page access control

**Modify: `agent/internal/handlers/pages.go`**

Update `Get()` handler with new access tier requiring both layers:

```go
func (h *PagesHandler) Get(w http.ResponseWriter, r *http.Request) {
    slug := chi.URLParam(r, "slug")
    keyID := KeyIDFromRequest(r)
    meta := readPageMeta(slug)

    // Tier 1: Paired user → any page (no code needed)
    if h.isPairedUser(keyID) {
        h.servePage(w, slug, meta)
        return
    }

    // Tier 2: Shared page — requires BOTH email match AND correct access code
    if len(meta.AllowedEmails) > 0 && meta.AccessCode != "" {
        email := IdentityEmailFromRequest(r)
        code := AccessCodeFromRequest(r)
        if email != "" && meta.isEmailAllowed(email) &&
           subtle.ConstantTimeCompare([]byte(code), []byte(meta.AccessCode)) == 1 {
            h.servePage(w, slug, meta)
            return
        }
    }

    // Tier 3: Public page → anyone
    if meta.Public {
        h.servePage(w, slug, meta)
        return
    }

    // Reject
    writeJSON(w, http.StatusNotFound, map[string]string{"error": "page not found"})
}
```

Note: `subtle.ConstantTimeCompare` for the access code to prevent timing attacks.

func (m *pageMeta) isEmailAllowed(email string) bool {

            return true

### Phase 3: Browser — Identity Token, Access Code Prompt & Storage

**Files to modify:**

#### 3.1 Shell.html — page viewing flow

**Modify: `web/public/p/shell.html`**

After discovery and before the E2E relay request:

```javascript
// 1. Fetch identity token (best-effort, fails silently for anonymous)
let identityToken = null;
try {
    const resp = await fetch('/api/identity-token', { credentials: 'include' });
    if (resp.ok) {
        const data = await resp.json();
        identityToken = data.token;
    }
} catch (e) {}

// 2. Load or prompt for access code
//    Store in IndexedDB keyed by "{username}/{slug}" so it's remembered per-page.
//    On first visit to a shared page: agent returns 404 (no code sent yet).
//    Shell shows a prompt: "This page requires an access code."
//    After entry, code is saved and request is retried.
let accessCode = await loadAccessCode(username, slug);  // from IndexedDB

// 3. Build inner request with both headers
const headers = {};
if (identityToken) headers['X-Identity-Token'] = identityToken;
if (accessCode) headers['X-Access-Code'] = accessCode;

const innerReq = {
    method: 'GET',
    path: `/api/page/${slug}`,
    headers: Object.keys(headers).length > 0 ? headers : undefined
};

let pageData = await encryptedRelay(agentUUID, aesKey, userKeyId, innerReq);

// 4. If rejected (404) and user is logged in, prompt for code
if (!pageData && identityToken && !accessCode) {
    accessCode = await promptAccessCode();  // modal/overlay input
    if (accessCode) {
        await saveAccessCode(username, slug, accessCode);  // IndexedDB
        headers['X-Access-Code'] = accessCode;
        innerReq.headers = headers;
        pageData = await encryptedRelay(agentUUID, aesKey, userKeyId, innerReq);
    }
}
```

Pages support three access levels:

- **Private** (default): Only paired users (page owner) can view

## Migration

- No database migrations needed (API or agent)
- `page.json` is backward-compatible — `allowed_emails` and `access_code` are optional, default to empty
- Agents without the verifier update will simply ignore the `X-Identity-Token` and `X-Access-Code` headers (they're inside the decrypted inner request, which existing code treats as an opaque HTTP request)
- Shell changes are backward-compatible — the identity token fetch and code prompt are best-effort
- IndexedDB version bump for `access-codes` store (existing stores preserved)

## Future Extensions

- **Wildcard domains**: `"allowed_emails": ["*@company.com"]` — allow all users from a domain
- **Named groups**: `"allowed_groups": ["team-alpha"]` — agent-local group definitions
- **Code rotation**: OpenClaw or agent can regenerate `access_code` to revoke access without changing `allowed_emails`
- **Revocation**: If the API needs to revoke a signing key, rotate `IDENTITY_SIGNING_KEY` and publish new JWKS. Agents refresh automatically.
