## Design Context

### Users
Technical early adopters — developers and power users who understand encryption, WebSocket, and self-hosting. Comfortable with terminal output and pairing codes. They value transparency about how things work and expect tools that respect their intelligence. They arrive from the OpenClaw/AI-agent ecosystem and want instant publishing without giving up control of their data.

### Brand Personality
**Friendly, simple, secure.** Approachable and casual but trustworthy under the hood. Think Notion's ease meets ProtonMail's privacy promise. Not intimidating, not dumbed down. The brand should feel like a capable friend who happens to be great at encryption.

### Emotional Goals
All three, in this priority:
1. **Confidence & control** — "My data is mine, I understand what's happening." Trust through transparency.
2. **Delight & ease** — "That was surprisingly easy." The magic moment when a page goes live instantly.
3. **Power & capability** — "I can do serious things with this." Professional tool, not a toy.

### Aesthetic Direction
- **Visual tone:** Sleek, modern, polished dark mode. Subtle and premium, like Vercel and Raycast. Clean information hierarchy, restrained use of color, purposeful micro-interactions.
- **References:** Vercel (deploy flow, dark UI, type hierarchy), Raycast (polish, keyboard-first feel, dark palette).
- **Anti-references:** Generic AI landing pages (gradient text, glassmorphism, hero metrics dashboards, card grids with emoji icons). Also avoid: cluttered dashboards, over-animated interfaces, "enterprise" gray-everything design.
- **Theme:** Dark mode only. The GitHub Dark palette (#0d1117 base, #161b22 surface, #e6edf3 text) is the foundation — it's already consistent and proven. No light mode planned.

### Color Palette
Already established and consistent across the codebase (GitHub Dark-inspired):
- **Backgrounds:** `#0d1117` (base), `#161b22` (surface/cards), `#21262d` (elevated/buttons)
- **Text:** `#e6edf3` (primary), `#8b949e` (secondary), `#656d76` (muted)
- **Borders:** `#30363d` (default), `#21262d` (subtle)
- **Accent:** `#58a6ff` (links, focus rings, interactive elements)
- **Semantic:** `#3fb950` (success), `#f85149` (danger), `#d29922` (warning), `#d2a8ff` (shared/purple)

### Typography
- **Primary font:** `'DM Sans', system-ui, -apple-system, sans-serif` — geometric but warm, loaded via Google Fonts with `display=swap` and preconnect. Matches "friendly, simple, secure" personality.
- **Monospace:** `'JetBrains Mono', ui-monospace, 'SF Mono', monospace` — for code elements, pairing codes, DNS records. Applied globally via `code, kbd, pre` selector.
- **Font smoothing:** `-webkit-font-smoothing: antialiased` on body for crisp dark-mode rendering.
- **Type scale (7 content stops):**
  - `0.75rem` — badges, timestamps, tiny metadata
  - `0.85rem` — button labels, secondary text, code snippets
  - `1rem` — body text, descriptions, form inputs
  - `1.15rem` — subheadings, section titles, dialog headings
  - `1.5rem` — page-level titles (dashboard h1, header logos)
  - `2rem` — landing section headings, display titles
  - `3.5rem` — hero display only
  - Compact dashboard text may use `0.8rem` or `0.9rem` where density is intentional.
- **Weights:** 400 (body), 600 (labels/subheadings), 700 (section headings), 800 (hero display only). No weight 500.
- **Line-heights:** 1.2 (headings), 1.6 (body/prose). Dark-on-light benefits from generous spacing.
- **Letter-spacing:** -0.03em on hero, -0.02em on h1/h2 headings. Default everywhere else.

### Design Principles

1. **Transparency builds trust.** Show what's happening (encryption status, agent connectivity, relay state) rather than hiding complexity. Users are technical — respect that. But present information clearly, not verbosely.

2. **Fewer elements, more meaning.** Every UI element should earn its place. Prefer information density over empty space, but never clutter. If something can be removed without losing function, remove it.

3. **Dark mode is the product.** This isn't a "dark theme option" — the dark palette IS the identity. Design for it natively. Ensure contrast ratios are strong, colored elements pop, and the interface feels intentional in dark.

4. **Instant feedback, zero friction.** Pages go live instantly. Pairing is one code. Deleting is one click + confirm. Match the product's speed in the UI — loading states should be fast, transitions purposeful (0.15s), and dead-ends rare.

5. **Security is invisible when it works.** E2E encryption is the backbone, but users shouldn't feel like they're in a security tool. Encryption details surface on hover/docs, not in the main flow. The feeling is "this just works and happens to be private."

### Spacing & Layout
- **Spacing scale:** 0.5rem, 0.75rem, 1rem, 1.25rem, 1.5rem, 2rem, 2.5rem, 3rem, 5rem
- **Border radius:** 4px (badges), 6px (buttons/inputs), 8px (popups), 10-12px (cards/sections)
- **Max content width:** 960-1060px centered. Cards/sections: 1.5rem-1.75rem padding.
- **Responsive breakpoint:** 700px (single column below, multi-column above)
- **Grid patterns:** 2-column on dashboard (agents), 3-column on landing (use cases), 1-column mobile

### Component Patterns
- **Buttons:** `.yb-btn-secondary` (dark surface, light text) and `.yb-btn-danger` (transparent bg, red border/text). No primary filled buttons except CTA.
- **Modals:** Portal-based, 75% overlay, centered, max-width 420-520px, escape + click-outside to close, focus-trapped.
- **Inputs:** Dark bg (#0d1117), border #30363d, focus ring #58a6ff. No outline:none without replacement.
- **Status indicators:** Colored dots + sr-only text. Green=online, gray=offline, yellow=needs action.
- **Cards/sections:** #161b22 bg, 12px radius, 1px solid #21262d border (sometimes), 1.5rem padding.

### Styling Approach
All styles are **inline React style objects** + **CSS-in-JS template strings** injected via `<style>` tags. No external CSS files, no Tailwind, no component library. This is intentional — keeps the bundle minimal and dependencies zero. When adding styles:
- Use inline style objects for component-specific styles
- Use CSS strings (injected via `<style>`) for pseudo-classes (:hover, :focus-visible), media queries, and class-based patterns that need to be shared
- Use `!important` in CSS strings only when overriding inline styles (this is a necessary pattern, not an anti-pattern, given the inline-first approach)
